# CH32V003

RISC-V2A, 16KB Flash, 2KB SRAM, 48MHz.

> **Note**
> CH32V003 is a riscv32ec core, which is not supported by ofiicial Rust yet.
> ch32-rs team maintains a fork of Rust at <https://github.com/ch32-rs/rust>.
> You can check [Noxim's Blog](https://noxim.xyz/blog/rust-ch32v003/introduction/) for riscv32ec support.

## Debug support

1-wire debug.

```text
SWDIO <-> D1/DIO@MCU, PD1
GND   <-> GND@MCU
3V3   <-> 3V3@MCU (No need if you connect your MCU board with USB-C power supply)
```

- [x] Flash support
- [x] Memory dump support
- [x] Reset / Resume
- [x] Reg read/write support
- [x] Boot area (`START_MODE` in USER option byte) — see below

## Boot area and option bytes (`START_MODE`)

After reset, the core can start from **user Code FLASH** (normal app at `0x08000000`) or from **System FLASH** (WCH boot / library region), depending on bit **5** of the **USER** option byte (`START_MODE` in the reference manual).

**WCH-Link quirk:** the probe does **not** reliably apply `BootFrom` as a standalone command. `wlink` sends it **inside** an active fast-program session (after the write region is set and the flash algorithm is loaded), matching WCH-LinkUtility behaviour.

**`system-boot` without a `.bin`:** the tool halts the core, reads the first **`write_pack_size`** bytes from the start of code flash (1024 bytes on CH32V003), then runs one program session that issues `BootFrom` and re-writes that same region unchanged. That size is the **fast-program chunk** for this chip family in `wlink`, not a datasheet requirement for the option byte itself; it keeps behaviour aligned with normal flashing and avoids guessing a smaller erase/program window without probe documentation.
